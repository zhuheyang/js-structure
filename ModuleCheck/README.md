# 代码单元测试

自己写的一些模块代码, 每次都至少要测试一遍, 次数多了, 每次都要新建一些文件, 这就没意思了.
新建的代码测试文件, 大多都是短小精悍, 无非是希望输入一个变数, 然后输出对应的结果, 然后符合自己的预期.  
但项目的代码多起来之后, 老是这样子打的话, 实在繁琐, 当然可以自己写一些自动化文件, 每次自己写完一个模块(文件夹)的代码后,
自动创建对应的测试文件, 然后让自己将这些测试代码填进去, 但这样子来实在很繁琐.  
最终的结果很可能是自己写了一个简化版的karma或者mocha进来了.就跟不使用NodeJS的框架, 结果最后做出了一个简化版的Express一样, 
能够熟练使用别人做好的工具也是一种天赋.

## 熟悉一项新事物的流程

首先是有了这样子的需求, 然后自己手动尝试了许多, 发现了其不方便性, 然后寻求成熟的, 众人一齐解决的成熟方案, 用成熟方案对自己已有的项目进行练习.  
在练习的基础上, 去了解对应模块或框架背后的设计模式与思想, 尝试自己造一个轮子出来, 或者对其原理进行一定的深究, 辅以大量的代码练习熟悉.

```js
// 自己写一些简单的test例子, 当然很方面, 运行一次, 然后输出结果符合预期即可
// 但不利于项目的持续变大, 因为往往到了项目后期, 都是要将这么多的测试用例给串起来的
// 以便于项目的持续继承. travis.yml方面的知识了.

// 同时, 如果中途更改了一些代码的文件名, 很有可能导致一些摸不着头脑的依赖bug, 这时如果通过单元测试用例, 
// 可减少这类情况的发生, 同时进行模块测试, 或者更高层次的测试时, 得到结果也更加稳固
// 发生问题时也可快速排除某单元出错的概率, 而将问题专注于各个接口之间的沟通
```
